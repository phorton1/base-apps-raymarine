## Code fixes applied this session (navMate is currently live with both)

### Fix 1: Two-step route_waypoints shift (nmOpsDB.pm ~line 1111)

`route_waypoints` has `PRIMARY KEY (route_uuid, position)` with INTEGER positions.
A single `UPDATE SET position=position+n WHERE position>=?` causes immediate UNIQUE
constraint violations because SQLite checks row-by-row.

Fix: two-step shift through a large temporary offset:

```perl
my $TEMP = 2_000_000;
$dbh->do(
    "UPDATE route_waypoints SET position=position+? WHERE route_uuid=? AND position>=?",
    [$TEMP, $route_uuid, $insert_pos]);
$dbh->do(
    "UPDATE route_waypoints SET position=position+? WHERE route_uuid=? AND position>=?",
    [$n - $TEMP, $route_uuid, $TEMP + $insert_pos]);
```

### Fix 2: route_point + PASTE_NEW reuses source wp_uuid (nmOpsDB.pm ~line 1128)

For the route_uuid path in `_pasteDB`, `$fresh=1` was calling `_insertFreshWaypoint`
for route_point items — wrong. "Paste new" on a route point means "insert a new
sequence position (duplicate reference) pointing to the same underlying WP UUID",
not "create a new waypoint record". The testplan §2.14a confirms: RP1's WP UUID
should appear again at the new position, with "The underlying waypoint record is
unchanged."

Fix:
```perl
my $itype = $item->{type} // '';
if ($itype eq 'route_point')
{
    $wp_uuid = $item->{uuid};   # duplicate reference, same WP
}
else
{
    $wp_uuid = _insertFreshWaypoint($dbh, $coll_uuid, $item->{data}, $source);
}
```

## Behavioral discoveries this session

### Discovery 1: Rapid-fire test commands drop intermediate queue items

When multiple `api/test?...&cmd=N` requests arrive before onIdle processes them,
only the first and last are reliably executed — intermediate selections overwrite
each other in the nmTest queue. Fix: always wait for STARTED/FINISHED in the log
before firing the next command. No code change needed; just runbook discipline.

Added to runbook: "Fire one command at a time — wait for STARTED/FINISHED before
the next mark+fire."

### Discovery 2: §3.1 E80→DB UUID-preserving paste is no-op when UUID already in DB

When IsolatedWP1 (ce4e43181f01b3ae) was pasted to E80 (§3.0a), the E80 preserved
the source UUID rather than assigning a new B2-prefix UUID. So E80_WP UUID ==
IsolatedWP1 UUID. When §3.1 then paste-downloaded E80_WP to DB with PASTE
(UUID-preserving), the code found the existing DB record and did nothing (silent
no-op). The WP did NOT appear in DST.

Result: §3.1 PASSED_BUT. The UUID-preserving path ran without error; the "already
exists in DB" case was handled gracefully. But the test was written expecting an
E80-native B2-prefix UUID (which would require a WP created on the E80 directly,
not uploaded from navMate). This is a test-design gap, not a code bug.

Note for last_testrun.md: record §3.1 as PASSED_BUT with this explanation.
