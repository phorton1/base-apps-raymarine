### Key design invariants driving the implementation pass

**SS1.6 (new this review):** Route paste ALWAYS preserves waypoint UUID references.
No new waypoint records are EVER created as part of any route paste -- PASTE or PASTE_NEW,
to any destination. PASTE_NEW for a route means the route record gets a fresh UUID only;
its route_waypoints references point to the same WP UUIDs as the source. Pre-flight
Step 4 guarantees those WPs exist at the destination before execution begins.

**SS10.1 Step 4 (new this review):** Universal route dependency check runs before any
destination-specific paste logic. If any route is in the resolved clipboard, every
referenced WP UUID must either already exist at the destination or be present in the
current clipboard (and will be processed before the route). Applies to both DB and E80
destinations. Reject with specific error if any WP UUID is absent from both.

**SS12.1 ordering (scoped this session):** Waypoints/groups before routes -- E80-source
pastes to DB only. DB-source pastes: no ordering required (WPs already in DB).

**[C1] nmOpsDB.pm `_pasteItemsToCollection` -- fresh route branch (~line 593)**
When `$fresh` (PASTE_NEW), calls `_insertFreshWaypoint` for every route_point.

**[C2] nmOpsDB.pm `_pasteItemsToCollection` -- non-fresh E80-source route branch (~line 617)**

when building route_waypoints. By the time execution reaches routes, WPs already exist

**[C4] nmOpsDB.pm `_pasteDB` PASTE_BEFORE/AFTER -- fresh route branch (~line 907)**

### CRITICAL -- Missing SS10.1 Step 4

**[C5] nmOps.pm `_doPaste` -- Step 4 entirely absent**
After Step 3 (recursive check), before the E80/DB branch split, there is no route
dependency check. Must add:
- Scan `@resolved` for items where `$item->{type} eq 'route'`
- For each route, check every `$rp->{uuid}` in `$item->{route_points}`
- For DB destination: verify UUID exists in DB via `getWaypoint($dbh, $uuid)`, OR is
  present as a waypoint/group-member UUID somewhere in the resolved clipboard

Note: DB-source routes to DB pass trivially (WPs already in DB). E80-source routes to DB
with heterogeneous clipboard pass if the WPs are in the clipboard. Pure route clipboard
from E80 to DB passes only if all referenced WPs already exist in DB.

**[C6] nmOps.pm `buildContextMenu` -- SS10.10 guard covers PASTE only, not PASTE_NEW**
The guard comment explicitly says "suppress PASTE (not PASTE_NEW)". Under SS1.6, PASTE_NEW
for a route also preserves WP refs and therefore also requires those WPs to exist on E80.
Fix: extend the same missing-WP check to also suppress PASTE_NEW when any route's member
UUIDs are absent from E80.

### IMPORTANT -- Ordering enforcement (E80-source to DB only)

## Design ambiguity to watch for

The main ambiguity: for Step 4 (C5), the "in the current clipboard" check for DB destination.
When a heterogeneous E80-source clipboard contains both waypoints and a route, the route's
WP UUIDs may not be in DB yet but ARE covered by the waypoints in the clipboard. The check
must recognize this: a WP UUID is "covered" if it appears as the uuid of any waypoint or
group-member in the resolved clipboard, not just if it exists in DB. If this logic is
ambiguous during implementation, stop and clarify before proceeding.
