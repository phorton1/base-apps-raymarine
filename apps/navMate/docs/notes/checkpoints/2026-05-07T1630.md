### Key design decisions resolved in this session

**Classification is per-operation, not canonical.** No single "effective_type" exists
for a selection. Each operation (Delete, Copy/Cut, Paste) brings its own interpretation
lens. The matrices describe what pre-flight determines, not a prior classification step.

**Ancestor-wins resolution requires UI surfacing.** When resolution removes items the
user explicitly selected, a confirmation dialog must identify the absorbed items before
proceeding. Not a silent transformation.

**Route is an intact compound object.** Selecting a Route node implies the route record
and its entire ordered route_waypoints sequence travel together.

**E80 header nodes are valid Delete right-click targets.** Right-clicking E80 Routes
header, Groups header, My Waypoints, or Tracks header offers Delete on all contents
of that folder. Four explicit rules added to SS8.2.

**E80 Paste Before/After is route-points only.** The E80 does not reliably support
positional ordering for waypoints within groups or routes/groups within their folders.
Those are name-sorted by the E80 tree (display-layer only, not nmOperations). SS10.9
simplified to three route-point rows plus one catch-all rejection.

**E80 tree display sort order** added to todo.md [Soon]: Groups by name (My Waypoints
floated), Routes by name, Tracks by name, Waypoints within Group by name; numeric
digit suffix sorted numerically not lexically.

**Intra-clipboard and E80-wide name collision checks** are Steps 6 and 7 in SS10.2 —
hard-abort conditions that run before any destination-specific rule. Both checks are
fast in-memory operations.

**Recursive paste check** (invariant SS1.5) is Step 3 in SS10.1 — walks destination
ancestry upward; rejects if any ancestor UUID matches a clipboard item.

**PASTE_NEW for route points** now fully specified in SS7.3: creates new route_waypoints
reference rows to the same underlying waypoint UUIDs; no new waypoint records. "New"
means a new reference position, not a new waypoint identity.

**Version field management** added as SS12.4: db_version/e80_version behavior for New,
DB modification, PASTE_NEW to DB, successful E80 upload, refresh upload, download from
E80 (UUID-preserving and fresh-UUID), and Cut-from-E80 + Paste-to-DB. Versioning is
first-class, not secondary to implementation semantics.
