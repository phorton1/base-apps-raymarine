## §3.4 bug investigation — what was found

§3.4 operation: COPY [E80_WP] (BOCAS1, ce4e43181f01b3ae) from E80 panel, PASTE to DB at DST.

Result: silent no-op. PASTE STARTED/FINISHED with zero WPMGR commands, DB record unchanged.

The relevant code is in `apps/navMate/nmOpsDB.pm`, function `_pasteOneWaypointToDB` (~line 148).
It has three paths:
1. UUID not in DB → insert (works)
2. UUID exists, `_wpFieldsDiffer` returns false → return 'no_change' (silent)
3. UUID exists, fields differ → show conflict dialog, then replace or skip

In §3.4, BOCAS1's data in DB is IDENTICAL to BOCAS1 on E80 (uploaded in §3.1 without changes),
so path 2 fires and returns 'no_change'. No update, no error.

**The real question:** is 'no_change' correct when data IS identical, or should any UUID-preserving
paste from E80 always overwrite the DB record?

Patrick's stated design intent: "copy the contents of the E80 WP to the given uuid in the DB" —
i.e., sync/overwrite field values regardless. Collection_uuid (tree location) must NOT change.

**Implication:** if Patrick means "always overwrite even when data is identical," the fix is to
remove or bypass the `_wpFieldsDiffer` check for E80-sourced pastes. If he means "only overwrite
when data differs," then path 2 is correct and the test setup was the problem (identical data
can't demonstrate a sync).

**Second issue to investigate:** path 3 (data differs → conflict dialog). With suppress=1, does
`_resolveConflict` auto-accept? If not, the conflict dialog would block in a real E80 sync
scenario. Look at how suppress interacts with `_resolveConflict`.

**My assessment:** §3.4 with identical data is likely correct behavior (no_change). The testplan
entry I wrote calling it a "known bug" may be wrong. Need Patrick's input after reading this.

Also check: `_wpFieldsDiffer` — does it compare the right fields? E80 WPs may lack some DB
fields (e.g., ts_source, db_version). If those cause spurious "differs" it could be a problem.
