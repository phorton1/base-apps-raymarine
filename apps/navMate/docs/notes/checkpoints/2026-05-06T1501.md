## Bug fix: deleted items not removed from Leaflet map

When a visible item (waypoint, route, track) was deleted via the context menu,
it was correctly removed from the navMate SQLite DB and the tree was refreshed,
but the item remained visible on the Leaflet map.

### Root cause

All delete paths in nmOpsDB.pm called `_refreshDatabase()` (defined in nmOps.pm)
which reloads the winDatabase tree but does nothing to the Leaflet map.
The `%rendered_uuids` hash in winDatabase.pm (which tracks what is currently
rendered) was never cleaned up, and `removeRenderFeatures` was never called.

### Fix (all changes committed in this session)

**winDatabase.pm** — added `onObjectsDeleted(@uuids)`:
Removes each UUID from `%rendered_uuids` and calls `removeRenderFeatures`.

**nmOps.pm** — added `_refreshDatabaseWithDelete(@uuids)`:
Calls `$pane->onObjectsDeleted(@uuids)` then `$pane->refresh()`.

**nmOpsDB.pm** — updated five delete paths to use `_refreshDatabaseWithDelete`:
- `_deleteDatabaseWaypoints` — collects UUIDs from `@$nodes`
- `_deleteDatabaseRoutes` — collects UUIDs from `@$nodes`
- `_deleteDatabaseTracks` — collects UUIDs from `@$nodes`
- `_deleteDatabaseBranch` — calls `getCollectionWRGTs` BEFORE `deleteBranch`
  to collect all descendant WRT UUIDs, then passes them to the delete helper
- `_deleteDatabaseGroupsAndWPs` — collects WP UUIDs from `%group_wps` before deleting

Non-delete paths (new items, paste, cut, group-delete-without-WPs,
`_deleteDatabaseCollection`) intentionally left using `_refreshDatabase()`.

### Not yet fixed

The cut path (`_cutDatabaseWaypoint`, etc.) has the same gap — cutting a
waypoint removes it from the DB but leaves it on the map. Deferred.
