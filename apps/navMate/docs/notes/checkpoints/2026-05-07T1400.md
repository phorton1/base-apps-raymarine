## Active design discussion — UNRESOLVED

### The effective_type formalization problem

Table column headers in §8.2 (and all similar tables) currently say "Type" — an informally
introduced term being used in a load-bearing way. It needs to become a formally defined
concept before it can appear in the operational sections.

### The cross-hierarchy selection gap

Root problem: COPY/CUT accept any selection, so the clipboard's `items` can contain objects
at multiple levels of the hierarchy simultaneously (e.g., a Group AND some of its member
Waypoints). The document defers classification to paste time but the classification logic
for cross-hierarchy cases has never been formally defined. This must be resolved before
`effective_type` (or whatever we name it) can be formalized.

**The fork:**
- **Normalize at COPY/CUT time** — collapse cross-hierarchy selections by a rule
  (ancestor wins? deepest-level wins?). Clipboard always holds items at a single
  consistent level. Makes effective_type cleanly definable.
- **Evaluate at paste time** — accept raw selection into `items` as-is; pre-flight
  classifies or rejects based on destination context. Complexity propagates through
  the whole system.

Patrick had not yet chosen a direction when checkpoint was taken.
