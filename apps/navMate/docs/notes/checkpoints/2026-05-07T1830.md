### Pre-flight is the central engine (SS5, SS8-SS10)

Pre-flight runs before EVERY operation (Delete, Copy, Cut, Paste alike). It does
NOT run inside Copy/Cut -- those are maximally permissive (non-empty selection only).
All classification is deferred to paste pre-flight where destination is known.

Critical pre-flight steps (SS10.1-SS10.2) for E80 destinations:
1. Ancestor-wins resolution (SS6.2) -- absorbs descendants; may fire confirmation dialog
2. Empty clipboard guard
3. Recursive paste check (SS1.5) -- walk destination ancestry; reject if UUID match
4. Branch dissolution (SS6.3) -- E80 only; branches have no E80 representation
5. Homogeneity check -- E80 only; heterogeneous clipboard = no paste commands offered
6. Intra-clipboard name collision -- hard abort
7. E80-wide name collision -- hard abort

### The clipboard (SS4)

Raw snapshot: {source: DB|E80, cut_flag: 0|1, items: [nodes as-captured]}
No normalization at copy/cut time. Ancestor-wins resolution deferred to paste.
Clipboard persists until replaced; paste does not clear it.

### Key behavioral rules to implement correctly

**DB cut to E80 is NEVER valid** (SS9) -- enforced at paste pre-flight, not at cut time.

**Route dependency check** (SS10.10) -- before ANY route paste to E80, every member
WP UUID must already exist in WPMGR. No silent side-effect WP creation.

**Execution guarantee** (SS5) -- once pre-flight passes, run to completion. Only
genuine E80 protocol failure permits partial outcome.

**E80 header nodes are valid Delete targets** (SS8.2) -- right-click on
header:groups, header:routes, header:tracks, or my_waypoints offers Delete on
all contents of that folder. Four explicit rules in SS8.2.

**PASTE_BEFORE/AFTER on E80 is route-point-only** (SS10.9) -- E80 does not support
positional ordering for WPs within groups or routes/groups within folders. Any
non-route-point destination on E80 panel rejects before/after paste.

**Version fields** (SS12.4) -- db_version/e80_version maintained on every write path.
db_version=1/e80_version=NULL on new item; db_version++ on modification;
e80_version=db_version on successful upload. Full table in SS12.4.

**Progress dialog** (SS12.5, SS13.4) -- all Paste-to-E80 and Delete-from-E80 operations
use the same ProgressDialog pattern as _doRefreshE80Data in winMain.pm. Do not reinvent.
