## Changes applied

### Schema version bump (a_defs.pm)
`$SCHEMA_VERSION` bumped from `'9.0'` to `'10.0'`.

### $db_def additions (c_db.pm)
`position REAL NOT NULL DEFAULT 0` added as final column to four table definitions:
`collections`, `waypoints`, `routes`, `tracks`.
`route_waypoints` unchanged — it already has `position INTEGER` for intra-route ordering.

### Migration block 9.0 → 10.0 (c_db.pm openDB)
New `if ($stored eq '9.0')` block:
1. `ALTER TABLE` adds `position REAL NOT NULL DEFAULT 0` to the four tables (existing rows land at 0).
2. Four correlated UPDATE statements initialize `position` from rowid order within each
   sibling scope — assigns 1, 2, 3, … preserving the import's accidental rowid order exactly.
   Collections scope: same `parent_uuid` (NULL-safe). WPs/routes/tracks scope: same `collection_uuid`.
3. Writes `'10.0'` to `key_values`.

Initial values are integers stored as REAL — correct and expected. Fractional values appear
only when mid-sequence insertion (Paste Before/After) is implemented later.

### ORDER BY queries updated (c_db.pm)
Seven queries changed from `ORDER BY rowid` to `ORDER BY position`:
- `getCollectionChildren` (2 queries — parent IS NULL and parent=?)
- `getCollectionObjects` (3 queries — waypoints, routes, tracks within collection)
- `getGroupWaypoints` (1 query)
- `getCollectionGroups` secondary sort: `w.rowid` → `w.position`

### Getter queries updated to return position (c_db.pm)
`position` added to SELECT list in: `getCollection`, `getTrack`, `getWaypoint`, `getRoute`.

### Detail panel display (winDatabase.pm)
`_fmt('position', ...)` added to `_showCollection` and all three branches of `_showObject`
(track, waypoint, route). Position appears after `collection_uuid` in each case (after
`comment` for collections).
