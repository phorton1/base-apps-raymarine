## Major Design Decision: nmOps Scheme Excision and Redesign

### Root cause

The nmOps scheme (nmOps.pm, nmOpsDB.pm, nmOpsE80.pm, nmClipboard.pm and
all the docs and tests around it) was designed for a phantom user: a boat
captain managing multiple chartplotters with pre-declared named working sets
of data. This is not Patrick's use case. Patrick has one boat, one region,
one E80, and loads it piecemeal with specific intentional items as needed.

Consequences of the phantom user model:
- Type-filtered copy/cut variants exist solely to manage heterogeneous
  clipboard contents being pushed wholesale to the E80
- The working_sets and working_set_members schema tables are completely unused
- Heterogeneous paste-to-E80-root was a half-baked implicit working set push
- nmUpload.pm was its explicit predecessor, now obsolete
- _e80_dedup.pm and nmOldE80.pm are obsolete for separate reasons
- 28 files across the codebase reference "context_menu"
- The context_menu_testplan.md and last_testrun.md represent many hard days
  of implementation and testing work on a system now being retired

### Excision plan (Commit 1)

Deleted outright (obsolete, no reference value):
- nmUpload.pm
- _e80_dedup.pm
- nmOldE80.pm

Moved to navMate/junk/ (reference value during redesign, NOT deleted):
- nmClipboard.pm
- nmOps.pm
- nmOpsDB.pm
- nmOpsE80.pm

All copy/cut/paste/upload handlers unwired from winDatabase.pm and winE80.pm.
winDatabase.pm _buildContextMenu survives, stripped to: Show/Hide on Map,
Delete, New Branch/Group. winE80.pm context menu removed entirely (temporary).
No new nmContext.pm needed.

### What the redesign builds

A simpler model for Patrick's actual use case:
- Copy = grab selection, no type filtering
- Paste to DB = permissive, UUID-keyed, no name dedup needed
- Send to E80 = targeted, intentional, item-specific operations
- Sync / Upload Newer = data-driven via db_version vs e80_version comparison
  (columns exist in schema 9.0; increment wiring is a separate todo item)

The versioning columns become the foundation for smart E80 sync operations
rather than an afterthought on the phantom working-set model.

### What is unaffected

NET protocol layer, c_db.pm, DB schema, Leaflet rendering, visibility system,
winDatabase.pm editor, Show/Hide on Map, Delete, New operations — all solid.
