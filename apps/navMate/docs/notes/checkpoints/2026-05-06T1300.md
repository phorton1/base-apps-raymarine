## Schema design decisions (this session)

### Versioning architecture: `db_version` + `e80_version` + `kml_version`

Added to `waypoints`, `routes`, and `tracks` (not collections, not route_waypoints):

- `db_version INTEGER NOT NULL DEFAULT 1` — bumped on every navMate edit; starts
  at 1 on insert. NOT YET wired up to increment — columns exist with correct
  defaults only. Full increment logic deferred to a future session.
- `e80_version INTEGER` — NULL = never synced to E80; set to `db_version` at
  time of successful upload/download. Not yet implemented.
- `kml_version INTEGER` — NULL = never exported to KML in the versioned sense.
  Not yet implemented.

### Design decisions recorded (for design_vision.md)

- Transport-specific columns baked into core tables — a conscious choice, not
  an oversight. Alternative (junction table `sync_state(object_uuid, transport,
  db_version_at_sync)`) rejected in favor of simplicity given the small,
  slow-moving transport list.
- Version numbers are NOT stored on the E80 (E80 can't hold them). At initial
  connect, `e80_version` is initialized from a token encoded in the E80 comment
  field (convention TBD — pending E80 character set testing and length limit
  verification). A waypoint arriving from E80 with no token has `e80_version=0`.
- Magenta display state (E80 newer than DB) is detectable via MODIFY events in
  d_WPMGR memory during a live session. At startup, a mismatch between the
  comment-encoded version and the field values reveals offline E80 modification.
- Tracks are uploadable (FSH file-based, not yet implemented) and downloadable
  (already supported). They carry all three version columns.
- `db_version` visibility changes (visible/invisible toggle) do NOT bump
  db_version — visibility is a display preference, not a syncable data change.

### Schema 9.0 migration runner

`openDB` in `c_db.pm` now detects old schema versions and migrates in place:
- If stored == '7.0': adds `visible` to all 4 tables, then adds version columns
  to waypoints/routes/tracks, updates key to '9.0'
- If stored == '8.0': adds version columns only, updates key to '9.0'
- If stored == '9.0': proceeds normally
- Any other version: existing mismatch error path (reimport required)

Migration runner is intentionally single-step and not a chain. Once the migrated
navMate.db is committed to git as the new 9.0 baseline, the migration branches
become dead code and can be removed on the next touch of c_db.pm.

The emergency backdoor remains: delete navMate.db before startup → fresh schema
created → oneTimeImport rebuilds from KML as always.

## wxPerl compatibility fix (this session)

Patrick's wxPerl does not have `SetItemState`, `GetItemState`, or
`SetStateImageList`. Replaced throughout `winDatabase.pm` with the normal image
list approach:
- `SetStateImageList` → `SetImageList`
- `SetItemState($item, N)` → `SetItemImage($item, N-1)` (0-based)
- `GetItemState($item)` → `GetItemImage($item)`
- `wxTREE_HITTEST_ONITEMSTATEICON` → `wxTREE_HITTEST_ONITEMICON`
- State mapping adjusted: unchecked=0, checked=1, indeterminate=2 (was 1,2,3)
- `Wx::NullBitmap` → `wxNullBitmap`

Migration ran successfully on the 7.0 baseline DB. navMate compiled and started.
