## Design Vision: winDatabase Multi-Editor

A batch-edit capability for the winDatabase editor when multiple items are
selected. No code yet — design conversation only. Add to design_vision.md
as a named cluster.

### Use cases

- Change color across a multi-selection of waypoints, routes, tracks
- Change comment across a diverse selection
- Change wp_type across a selection of waypoints
- Name and lat/lon deliberately excluded — no useful batch semantic

### Inventory multi-editor pattern (reference implementation)

Patrick has a working multi-editor in `/base/apps/inventory` (JavaScript).
Key design decisions from that implementation that apply here:

- **Mixed-value indicator**: placeholder text "(Multiple Values)" in text
  fields — not a color discriminator. Blank/gray is insufficient; the
  placeholder text makes the mixed state legible.
- **Changed indicator**: field background color shows which fields have been
  touched and will be written on save. User always has a clear read of
  exactly what will propagate to all selected items.
- **Error indicator**: field color for validation errors (e.g., name too
  long). Not a navMate concern yet but the pattern should be kept in mind.
- **Only touched fields are written**: the core contract. Fields the user
  did not touch are never written, even if visible. Requires per-field
  dirty tracking separate from the global editor dirty flag.

### Color swatch problem

"(Multiple Values)" placeholder text does not translate to a color swatch.
Open design question: hatched/striped swatch? Disabled swatch + adjacent
text label? "Pick..." button still active (picking sets a new color for all)?
No decision yet — leave as an open item within the cluster.

### Pub::Database sparse update support

`Pub::Database::update_record` accepts sparse input hashes — only the fields
passed are written. This resolves the DB-layer side of the "only write touched
fields" problem. The editor still needs per-field dirty tracking to know which
fields to include in the hash, but the DB layer will not stomp untouched fields.
