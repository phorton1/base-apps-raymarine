## What was established about §3.11

The §3.11 FAIL was a test-data selection error compounded by a conceptual carryover from
the old context_menu system. Navigation/Routes was the wrong branch to use because it
contains both route objects and group objects. The homogeneity check (SS10.2 Step 5)
correctly blocked that paste. The "ordering dependency" concept in §3.11 -- groups before
routes within a single E80 paste -- is a legacy concept that no longer applies because
the homogeneity check makes mixed-type E80 pastes impossible. §3.11 needs to be retired
or reconceived entirely. The proposed fix in last_testrun.md ("expand permitted two-type
mixes to include group+route") is wrong.

The 3.0c bug (`$item->{members}` instead of `$item->{route_points}` in `_pasteRouteToE80`)
is a separate issue, a code-level manifestation of the same conceptual carryover. It also
explains the §5.6 FAIL.

## The broader concern

Patrick identified that I contaminated the artifacts of the rewrite (testplan, code) with
old-design assumptions. The whole point of the rewrite was to sacrifice mixed-type E80
pastes in exchange for clean design. The contamination showed up in two places we know
about: §3.11 in the testplan, and the `members`/`route_points` bug in the code. Patrick
directed a full top-down review: design first, then implementation, then testplan, then
runbook.

## Design review of nmOperations.md -- completed this session

A thorough conversation-driven review of the design document was completed. The following
changes were made to nmOperations.md:

### New invariants added (Section 1)

**SS1.2 strengthened:** Route dependency check is now stated as universal (all
destinations, PASTE and PASTE_NEW alike). Explains enforcement differs by side: trivially
satisfied for DB-source pastes; explicitly checked by pre-flight Step 4 for E80-source
pastes.

**SS1.6 (new):** Route paste always preserves waypoint UUID references. No new waypoint
records are ever created as part of any route paste. PASTE_NEW for a route means the route
record gets a fresh UUID; the waypoint references it carries are unchanged.

**SS1.7 (new):** E80 paste operations must be type-homogeneous. Deliberate design choice.
One exception: mixed waypoints-and-route-points at E80 route point destinations.

### Section 7 -- new SS7.4

Documents route compound paste behavior: route paste always UUID-preserves its waypoint
references; pre-flight must confirm all referenced WPs exist at destination or in the
operation; member waypoints in the selection are handled independently and do not affect
the route's own references.

### Section 10 -- pre-flight changes

**SS10.1 Step 4 (new):** Universal route dependency check, all destinations. If the
clipboard contains route items, all referenced waypoint UUIDs must exist at destination or
be in the current clipboard. Applies before any destination-specific rule.

**SS10.2 steps renumbered:** Former Steps 4-7 are now Steps 5-8 to accommodate the new
SS10.1 Step 4.

**SS10.3 table:** Routes PASTE_NEW note corrected ("fresh route UUID; waypoint refs
preserved"). Two rows added for heterogeneous E80-source clipboards (PASTE and PASTE_NEW).

**SS10.10:** Route dependency check text removed (now in SS10.1 Step 4). SS10.10 now
covers only UUID conflict resolution. Step references updated to 7 and 8.

### Section 12 -- semantics changes

**SS12.1:** Two major additions:
- "Replace means update in-place" -- existing UUIDs are updated where they are;
  collection_uuid is never changed; paste destination is landing zone for new items only.
- Execution ordering -- waypoints and groups always before routes; ensures any
  user-initiated Abort cannot leave broken route references in the DB.

**SS12.3:** Route PASTE_NEW corrected. Removed "each member waypoint also receives a fresh
UUID." Now states: fresh route UUID, waypoint refs preserved, no new waypoint records.

**SS12.5 (upload):** Dead-code bullet removed ("Branch dissolved: groups and their
waypoints first, then routes. Track items skipped silently."). Replaced with correct
statement: effective contents at execution are always homogeneous (Step 6 already blocked
heterogeneous).

**Numbering fix:** Duplicate §12.5 resolved. Second §12.5 (Paste Before and After) is now
§12.6. Delete -- Database is §12.7. Delete -- E80 is §12.8.

## Key design decisions made this session (for implementation review)

1. Route paste is always UUID-preserving for its waypoint references -- no exceptions
2. PASTE_NEW for a route = fresh route UUID only; same waypoint UUID refs
3. E80->DB paste: existing UUIDs updated in-place, never re-parented
4. E80->DB paste: waypoints/groups always processed before routes
5. Heterogeneous E80-source clipboards are valid for DB-destination paste
6. The homogeneity check is E80-destination-only and is correct as written
7. §3.11 ordering concept is dead -- the homogeneity check makes it unreachable
