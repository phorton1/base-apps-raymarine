## Design decisions reached -- must propagate to nmOperations.md

### 1. The core UUID constraint (clarified)

COPY+PASTE (UUID-preserving) within the same DB always produces duplicate UUIDs.
This is the one hard logical constraint. It applies to ALL types -- waypoints,
groups, routes, tracks. There is no type-specific exception.

The only UUID-preserving DB->DB operation is CUT+PASTE (move). UUID is preserved,
record is unique.

### 2. PASTE_NEW to DB is always valid

Any source (DB or E80), any type (WP, group, route, track). Fresh UUID, lands
exactly where the user right-clicked. No logical restriction. The current design
doc prohibition of PASTE_NEW for track clipboards (SS10.3 / SS12.3) is wrong
and must be removed.

Track's only specialness: tracks can only be originally created on the E80.
Once in the DB they are records like any other -- fresh-UUID copy is coherent.

### 3. E80->DB PASTE (UUID-preserving, non-cut) is the hairball

Two completely different behaviors depending on hidden UUID state:
- UUID absent from DB: insert at right-clicked destination (classic paste)
- UUID present in DB: would update existing record in-place; right-click
  destination is silently ignored -- the operation has nothing to do with
  where the user clicked

Multi-item clipboard compounds it: a single PASTE gesture could have some
items landing at destination and others updating in-place at scattered DB
locations. Non-uniform outcome from one gesture.

### 4. Resolution: PASTE reclassified; SYNC is a new CTX_CMD

At COPY/CUT time from E80, immediately classify the clipboard by checking
DB UUID existence:

- ALL UUIDs absent from DB: clipboard is a "paste" -- offer PASTE in context menu
- ALL UUIDs present in DB: clipboard is a "sync" -- offer SYNC in context menu,
  disable PASTE
- MIXED: disable both PASTE and SYNC; PASTE_NEW remains available;
  status bar explains why (e.g. "Paste/Sync not available: clipboard contains
  both new and existing items -- use Paste New")

PASTE_NEW is always available regardless of classification.

### 5. SYNC as a new CTX_CMD (immediate, not clipboard-mediated)

SYNC is a direct in-place operation, like DELETE -- not a clipboard operation.
The user right-clicks selected item(s) that have UUID counterparts on the other
panel. SYNC pushes this side's version to the matching UUID on the other side.

Properties:
- New CTX_CMD constant (value TBD, after existing range)
- Available on both panels (DB->E80 and E80->DB)
- Multi-select: yes (select several mismatched items, right-click, SYNC)
- Preflight: "newer wins" confirmation if target version is newer
- Currently determinable by field comparison (lat/lon, name, color, etc.)
- Better with versioning/color scheme: blue=synced, red/green=older/newer;
  the color scheme is the conceptual foundation that makes SYNC comprehensible
- NOT a clipboard operation; does not consume or produce clipboard state

The color/versioning scheme is not just display -- it makes the PASTE vs SYNC
distinction visible to the user at all times. Without it, SYNC is a confusing
label. With it, the user already understands UUID correspondence before they act.

### 6. Clipboard clearing on DELETE and SYNC

**Rule: DELETE and SYNC both clear the entire clipboard unconditionally.**

No partial clearing. All or nothing.

Rationale: too many edge cases otherwise (e.g. UUID deleted from E80 does not
affect a DB-sourced clipboard item with the same UUID). Simple rule, no
rabbit holes.

**Current code gap:** clearClipboard() exists in nmClipboard.pm but is never
called from nmOps.pm's delete path. DELETE currently does NOT clear the
clipboard. Latent bug -- CUT item then DELETE it separately leaves a dangling
cut reference in the clipboard.

### §4.4 -- reclassify from FAIL to PASS (pending design doc update)

The test expected PASTE_NEW from E80 track clipboard to DB to be rejected
(per SS10.3 / SS12.3). That prohibition is now removed. What happened in
Cycle 9 §4.4 (fresh-UUID track created in DB) is correct behavior.
last_testrun.md §4.4 must be corrected when nmOperations.md is updated.

### §3.4 / §3.12 / §3.13 -- reclassify

These tested E80->DB PASTE where the UUID already exists in DB (PASSED_BUT,
known silent-no-op bug). Under the new design, the PASTE command is not
offered in this case -- the clipboard would be classified as "sync" and only
SYNC would appear. The test steps themselves need redesign for Cycle 10:
- §3.4/§3.12/§3.13 style: use PASTE_NEW (always valid) for the download path
- New §3.x: test SYNC command once implemented

### Guard tests §5.14 / §5.15 / §4.3 (silent rejection, no IMPLEMENTATION ERROR)

These are real bugs -- operations that should log IMPLEMENTATION ERROR but
fire silently. Unaffected by design changes. Still need code fixes.

### §5.9 (recursive paste guard bypass) and §5.16 (mixed-clipboard PASTE_BEFORE)

Still real bugs. Unaffected by design changes.
