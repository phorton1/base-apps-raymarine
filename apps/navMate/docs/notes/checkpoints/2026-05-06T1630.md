## Problem scope (broader than design_vision described)

The design vision noted the bug only in `_onShowMap`. The visibility checkbox
work exposed the same bug in `_pushCollectionToLeaflet`. Both call helpers that
call `addRenderFeatures` per object — each call increments the server version,
and Leaflet zooms to each version bump. The fix is the same in both places.

Root cause functions that call `addRenderFeatures` per-object:
- `_pushObjToLeaflet` — called in a loop by `_pushCollectionToLeaflet`
- `_renderCollection` / `_renderObject` — called in a loop by `_onShowMap`

Note: `_renderCollection` already accumulates all features internally and makes
one `addRenderFeatures` call — it is fine on its own. The problem is only when
multiple `_renderCollection` or `_pushObjToLeaflet` calls are made in sequence.

## Implementation plan (for primary Claude after /clear)

### Approach: optional accumulator parameter

Add an optional `\@acc` parameter to `_pushObjToLeaflet`, `_renderCollection`,
and `_renderObject`. When the caller passes an accumulator ref, the function
pushes features into it instead of calling `addRenderFeatures`. The top-level
caller collects everything and makes one call at the end.

### `_pushCollectionToLeaflet` — rewrite

```perl
sub _pushCollectionToLeaflet
{
    my ($dbh, $this, $uuid) = @_;
    my $wrgt = getCollectionWRGTs($dbh, $uuid);
    my @features;
    _pushObjToLeaflet($dbh, $this, { %$_, obj_type => 'waypoint' }, \@features) for @{$wrgt->{waypoints}};
    _pushObjToLeaflet($dbh, $this, { %$_, obj_type => 'route'    }, \@features) for @{$wrgt->{routes}};
    _pushObjToLeaflet($dbh, $this, { %$_, obj_type => 'track'    }, \@features) for @{$wrgt->{tracks}};
    addRenderFeatures(\@features) if @features;
}
```

### `_pushObjToLeaflet` — add optional accumulator

Change the final line from:
```perl
    addRenderFeatures(\@features) if @features;
```
to:
```perl
    if ($acc) { push @$acc, @features } else { addRenderFeatures(\@features) if @features }
```
and add `$acc = undef` as a 4th optional parameter.

### `_onShowMap` — collect across all nodes then one call

Refactor the loop to collect into `@all_features`, passing accumulator to
`_renderCollection` and `_renderObject`. Make one `addRenderFeatures` call
after the loop. This requires `_renderCollection` and `_renderObject` to also
accept an optional accumulator and push into it rather than calling
`addRenderFeatures` when one is provided.

### Leaflet zoom behavior

Each `addRenderFeatures` call triggers one version bump; Leaflet zooms to the
bounding box of the features in that call. Batching into one call means one
zoom to the collective bounding box — no Leaflet-side changes needed.
Verify by checking nmServer.pm `addRenderFeatures` and the Leaflet JS handler
before coding, to confirm the zoom behavior is version-bump-driven.

### `_pullCollectionFromLeaflet`

The remove path (`_pullFromLeaflet` called per-object) has a similar pattern
but the zoom issue doesn't apply on removal. Leave as-is for now; revisit if
there are visible flickers on large removals.
