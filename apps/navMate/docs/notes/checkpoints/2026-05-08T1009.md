### Question 1: Is PASTE_BEFORE/AFTER general for all anchor/item types?

Answer: No — it was waypoint-only. Fixed comprehensively.

**nmOpsDB.pm — `_pasteDB` collection-member path (the `if (!$route_uuid)` block):**

Completely rewritten. Previous version only handled a waypoint as the anchor
(called `getWaypoint` on any anchor UUID, failing silently for routes/tracks/groups).
Also only accepted waypoints/route_points from the clipboard (silently dropped others).

New version:

1. **Anchor type dispatch** — dispatches on `$anchor_type`/`$anchor_ot`:
   - `object/waypoint` → `getWaypoint` → `{collection_uuid, position}`
   - `object/route`    → `getRoute`    → `{collection_uuid, position}`
   - `object/track`    → `getTrack`    → `{collection_uuid, position}`
   - `collection`      → `getCollection` → `{parent_uuid, position}` (container = parent_uuid)

2. **Cross-table neighbor query** — UNION across all four item tables to find the
   true nearest occupied position regardless of item type:
   ```sql
   SELECT MAX/MIN(p) FROM (
       SELECT position AS p FROM waypoints   WHERE collection_uuid=? AND position</>?
       UNION ALL
       SELECT position AS p FROM routes      WHERE collection_uuid=? AND position</>?
       UNION ALL
       SELECT position AS p FROM tracks      WHERE collection_uuid=? AND position</>?
       UNION ALL
       SELECT position AS p FROM collections WHERE parent_uuid=?    AND position</>?
   )
   ```

3. **All clipboard item types handled** — waypoint, route_point, route, track, group,
   branch. Each follows the three-branch pattern (fresh / db-cut-move / uuid-preserving):
   - Routes: fresh inserts route_points then route, cut does `UPDATE routes SET
     collection_uuid=?, position=?`, uuid-preserving uses insertRouteUUID/updateRoute
   - Tracks: fresh inserts track+points, cut does `UPDATE tracks SET collection_uuid=?, position=?`
   - Groups/branches: fresh uses insertCollection, cut does `UPDATE collections SET
     parent_uuid=?, position=?` (the move IS the operation; no separate cut step)

4. **Cut semantics bug fixed** — previous version called `_cutDatabaseWaypoint` AFTER
   doing `UPDATE waypoints SET collection_uuid=?`, which would delete the already-moved
   item. Now: `cut_flag && source eq 'database'` does a pure UPDATE and skips @to_cut.
   Only E80-source cuts push to @to_cut.

**nmClipboard.pm — `getPasteMenuItems`:**
- Extended DB `$positional` to include all collection types (was group-only, now
  `$t eq 'collection'` covers both group and branch).

### Question 2: PASTE vs PASTE_BEFORE/AFTER semantics for route_points

Design issue: PASTE (non-NEW) on a route_point anchor with mixed clipboard content
(non-route_point items) leads to waypoints ending up in two collections. Also,
cut cleanup called `_cutDatabaseWaypoint` which refused deletion due to route refs,
leaving the source route unchanged.

**nmClipboard.pm — `getPasteMenuItems`:**
- Added `$allow_non_new` check: on a `route_point` anchor, PASTE_BEFORE/AFTER
  (non-NEW) is suppressed unless ALL clipboard items are route_points. Mixed content
  must use PASTE_NEW (fresh UUIDs, no ownership confusion).

**nmOpsDB.pm — route_point PASTE_BEFORE/AFTER path (the `if ($route_uuid)` path):**

Two fixes:

1. `@wp_items` flattening now includes `route_point` type (previously dropped them,
   causing "no waypoints to insert" failure — critical since with the new menu rule,
   the clipboard will contain route_points, not waypoints, when PASTE non-NEW is used).

2. Cut cleanup rewritten: DB-source route_points now call `removeRoutePoint(dbh,
   route_uuid, adjusted_position)` INSIDE the open DB connection. The adjusted
   position accounts for the shift already applied (`+n` if `orig_pos >= insert_pos`).
   The old `_cutDatabaseWaypoint` path (which always refused due to route refs) is
   skipped for DB-source route_points.

### Question 3: PASTE vs PASTE_BEFORE/AFTER availability on collection vs terminal nodes

PASTE was previously shown unconditionally (on all non-tracks nodes). This is wrong —
PASTE should only appear when the destination can receive items into itself (a collection).
PASTE_BEFORE/AFTER is the correct verb for terminal nodes (insert adjacent to, not into).

**nmClipboard.pm — `getPasteMenuItems` — added `$is_collection` flag:**

```
E80 collections (PASTE allowed): header/groups, header/routes, my_waypoints, group, route
E80 terminal (PASTE suppressed): waypoint, route_point
DB collections (PASTE allowed):  root, collection (branch or group)
DB terminal (PASTE suppressed):  object (waypoint/route/track), route_point
```

`$is_collection` gates PASTE/PASTE_NEW independently from `$positional` which gates
PASTE_BEFORE/AFTER. The two are now orthogonal:
- DB collection/branch: both is_collection AND positional → all six paste variants
- DB object/waypoint: positional only → PASTE_BEFORE/AFTER only
- E80 group: is_collection only → PASTE/PASTE_NEW only
- E80 waypoint: neither → no paste at all
