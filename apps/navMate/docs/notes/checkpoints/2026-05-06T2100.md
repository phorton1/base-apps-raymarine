## The New nmOps Design — Agreed Principles

### Core metaphor

Copy / Cut / Paste — same familiar verbs, radically simplified semantics.
No "Send to E80" menu item. Paste-to-E80 IS the upload path.

### Cut semantics (critical asymmetry)

- **Cut from DB** = DB-internal move ONLY. Re-homes `collection_uuid`.
  Never valid as a source for Paste-to-E80. DB is always the owner.
- **Cut from E80 → DB** = valid. Transfers ownership to navMate.
  The primary use case is tracks: record on E80, cut to DB.

### Homogeneous selection rule

If the clipboard selection is not homogeneous (same object type), Paste is
simply not offered. No attempt to handle mixed-type pastes.

**Exception:** a single selected collection (branch OR group) is a valid
paste source even though its contents are heterogeneous — the user selected
ONE thing, not a mixed bag. This covers the old Copy-ALL use case.

Copy-ALL as a separate operation disappears. Users select a branch to get
that behavior.

### Clipboard vocabulary (collapsed from old 36-state matrix)

Valid clipboard types:
  waypoint(s) | route(s) | track(s) | group(s) | collection/branch

Each with: source (db | e80), cut (0 | 1).

Heterogeneous multi-select → clipboard not set, Paste not shown.

### Pre-flight validation model

ALL validation happens before any E80 write operation begins.
Operations are all-or-nothing. No partial execution except for genuine
unforeseen E80 protocol failures (disconnect, unexpected NAK, etc.).

Pre-flight checks run in this order:
1. Route dependency check (for route pastes to E80): verify all member
   WP UUIDs exist in WPMGR memory. If any missing → error dialog listing
   the route(s) and missing WPs → abort. User must paste waypoints first.
2. UUID conflict check: categorize each item into:
   - Clean create: UUID not on E80 → proceed
   - UUID match, db_version > e80_version → refresh without question
   - UUID match, E80 may have independently changed → warn, user decides
   - Name collision (UUID not on E80, name exists on E80) → hard fail,
     no auto-renaming, no dialog-to-continue. Operation aborted.
3. Item count dialog (if applicable): "N items already exist on E80 —
   skip them and continue with the rest, or abort?"

### The route paste simplification (biggest single win)

Route paste to E80 NO LONGER carries waypoints.
- Old: paste route → secretly paste waypoints first → maintain group
  structure → resolve waypoint conflicts → then paste route. Three nested
  semantic levels inside one async operation.
- New: pre-flight checks WP UUIDs exist in WPMGR → if yes, CREATE route
  referencing existing E80 UUIDs → done. One clean operation.

The ordering dependency, the `no_change` idempotency logic, the waypoint
conflict resolution inside route context — all gone.

### Groups complexity (remains, but bounded)

Groups still have ordering complexity (group before members) and mixed
semantics with waypoints. This is tractable because it is self-contained —
groups don't secretly contain a third nested semantic level.

### Orphaned items

Items deleted from navMate but never propagated to E80 leave navMate UUIDs
on E80 with no DB record. E80 is their implicit owner in that state.
Pre-flight must not treat these as conflicts — they are simply items the
pre-flight UUID check won't find in the DB, not something navMate needs to
manage.

### Versioning (deferred layer)

db_version vs e80_version comparison is the foundation for smart sync.
The column wiring exists (schema 9.0) but increment wiring is deferred.
For now: UUID match → simple skip-or-abort dialog. The full versioning-aware
behavior (distinguishing "refresh stale E80" from "overwrite E80-modified")
is a dedicated future session.

### Testability requirement

The `$suppress_confirm` mechanism in `nmDialogs.pm` MUST carry forward and
be extended to cover BOTH confirmation dialogs AND error/warning dialogs.
This is required to test pre-flight failure paths programmatically (not just
success paths).

The `/api/test` HTTP endpoint machinery (nmTest.pm equivalent) must be
rebuilt for the new simplified command set.

The progress dialog protection pattern (`_doRefreshE80Data` in winMain.pm)
must be reused for any Paste-to-E80 operation. This is hard-fought
behavioral knowledge — do not reinvent.

`dialog_state` and `close_dialog` endpoints carry forward unchanged.
